⚠ MESHCORE SUPPORT — BETA
MeshCore is a separate mesh protocol from Meshtastic. MeshDash integrates it via the meshcore Python library over Serial, TCP, or BLE. This integration is in active development. Please read the capability and limitation notes below before connecting.
✓ WHAT WORKS
● Receive channel messages — all channels (0–7), displayed in the Channels view
● Receive direct messages — DMs to your node appear in the DMs view
● Send channel broadcasts — BROADCAST button works on all configured channels
● Send DMs — to any node that has been seen via advertisement or contacts list
● Node discovery — advertisements populate the node list and map
● GPS / map plotting — nodes with location data appear on the map
● Battery & telemetry — battery level, voltage, uptime, noise floor displayed in overview
● Firmware & radio info — firmware version, frequency, bandwidth, spreading factor displayed
● Packet statistics — RX/TX counts (flood vs direct) shown in analytics panel
● ACK delivery tracking — tick marks update to delivered on DMs
● Auto-advertisement — MeshDash announces your node on connect and every 10 min so others can reach you
● SNR/RSSI per-hop data — RX_LOG events correlated to messages, attached to packet log entries
● Path / traceroute data — PATH_UPDATE events translated to traceroute packets
● Multi-slot — add multiple MeshCore nodes alongside Meshtastic and MQTT slots
● Auto-reconnect — exponential backoff reconnection on disconnect
● Queued message replay — messages stored on radio while disconnected are delivered on reconnect
● Scheduler & auto-reply — scheduled messages and auto-replies work correctly via MeshCore sendText
✗ KNOWN LIMITATIONS
● DMs to completely unknown nodes may fail — a node must have advertised at least once while you were connected for its public key to be known. MeshDash mitigates this by broadcasting your own advertisement on connect and every 10 minutes, so remote nodes automatically add you to their contact list. Nodes that are offline and have never been seen cannot be reached.
● Channel messages may not arrive on some firmware versions — a confirmed bug in nRF52840 Companion USB firmware v1.11.x causes CHANNEL_MSG_RECV events to not fire from external devices. MeshDash works around this with a periodic get_msg() fallback poll, but reception may still be unreliable on affected hardware. Upgrade to the latest MeshCore firmware to resolve this.
● Channel sender names anonymous on pre-V3 firmware — older firmware embeds the sender name as "Name: message body" in the text. MeshDash attempts to resolve the sender via case-insensitive contact lookup; if no match is found, the sender appears as a stable anonymous node per channel. V3 firmware sends the pubkey prefix directly and resolves correctly.
● No node-to-node DM interception (by design) — unlike MQTT observer mode, MeshCore companion radio connections only see messages addressed to or from your node. Traffic between other nodes is not visible unless relayed through your radio. This is a fundamental MeshCore protocol design decision, not a MeshDash limitation.
● Device contact list is finite — MeshCore radios store 50–350 contacts depending on firmware version. Once full, new nodes cannot be added until old ones are pruned. MeshDash cannot currently manage the device contact list (add/remove contacts).
● Node IDs are 12 hex chars, not 8 — MeshCore identifies nodes by a 6-byte public key prefix, displayed as !a1b2c3d4e5f6. Meshtastic uses 8-char IDs. Cross-protocol DMs (Meshtastic ↔ MeshCore) are not supported — they use incompatible key systems.
● BLE transport is experimental — BLE pairing behaviour varies by OS. Linux requires prior device pairing via bluetoothctl before connecting. macOS handles pairing via the system Bluetooth UI. BLE proxy connections do not support PIN pairing. Range and stability depend on host Bluetooth hardware quality.
● Serial on pseudo-ttys (socat) fails — the MeshCore library sets RTS/DTR signals on connect. Socat pseudo-ttys do not support hardware flow control ioctls and will error. Use a real USB serial adapter (/dev/ttyUSB0, /dev/ttyACM0, COM3). All genuine MeshCore hardware (Heltec, LilyGO, RAK) works correctly.
● Admin commands not yet exposed in UI — the MeshCore protocol supports remote admin (login, radio config, set name/coords, reboot) via the library. MeshDash does not yet expose these in the interface. MeshMonitor has this feature; it is on the MeshDash roadmap.
